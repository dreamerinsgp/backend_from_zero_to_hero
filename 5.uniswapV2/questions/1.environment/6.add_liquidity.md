# Q1：已有 Pair 后如何加流动性？`mint` 需要什么前提？

本仓库 **`v2-core`** 里，向池子加流动性的入口是 **`UniswapV2Pair.mint(to)`**。它**不会**从 `msg.sender` 里 `transferFrom` 拉币，而是根据 **Pair 合约地址上两种代币的余额相对上次储备的增量** 来铸 LP。

---

## 机制概要（必须先理解）

1. **`mint(to)` 做的事**（见 `contracts/UniswapV2Pair.sol`）：  
   - 读当前 **`reserve0` / `reserve1`** 与 **`balanceOf(pair)`**；  
   - **`amount0 = balance0 - reserve0`**，**`amount1 = balance1 - reserve1`**（本次“多出来”的代币）；  
   - 按规则计算 **`liquidity`**，给地址 **`to`** 铸 LP（ERC20）；  
   - 更新储备。  

2. **因此**：在调用 **`mint`** 之前，你必须先把 **token0 / token1** 转到 **Pair 地址**上（余额要大于当前储备），否则 **`amount0`、`amount1` 为 0**，会 **`INSUFFICIENT_LIQUIDITY_MINTED`**。

3. **合约注释**写明：`mint` 是底层接口，**应由做了安全检查的合约调用**（生产环境通常是 **Uniswap V2 Router**，在 **`v2-periphery`** 里），自己直接调 `mint` 时要自己算好比例与滑点。

---

## 操作步骤（仅 core、手工流程）

假设 Pair 已部署，且你已知道 **`token0`、`token1`**（可用 `Pair.token0()` / `token1()` 查询；与创建时传入顺序无关，链上已排序）。

1. **准备钱包**：在对应网络上有 **原生币（gas）**，且持有足量 **两种 ERC20**。  
2. **把两种代币转入 Pair**（二选一常见做法）：  
   - 用户直接对 **`ERC20.transfer(pairAddress, amount)`** 各转一笔；或  
   - 若由**你自己的合约**代转，则先 **`approve`** 再 **`transferFrom`** 到 Pair。  
3. **调用 `pair.mint(to)`**：`to` 为接收 **LP 代币**的地址（通常是你自己的地址）。  

**首笔流动性**（池子原先储备为 0）：铸币量约为 **`sqrt(amount0 * amount1) - MINIMUM_LIQUIDITY`**，其中 **`MINIMUM_LIQUIDITY = 10^3`** 会永久锁到 **`address(0)`**，用来防止首铸攻击。你存入的 **比例** 会决定初始价格。  

**后续加流动性**：铸币量与 **`amount0/reserve0`、`amount1/reserve1`** 中较小的一侧一致；若只转了一种币或比例与池子不一致，另一种的“有效”增量会按公式取 **min**，可能造成不理想的比例，**生产上应用 Router 按公式算好再转**。

---

## 前提条件清单

| 前提 | 说明 |
|------|------|
| **Pair 已存在** | 已通过 **`Factory.createPair`** 创建并完成 **`initialize`**。 |
| **代币已到账 Pair** | 在 **`mint`** 交易执行前，Pair 的 **`token0` / `token1` 余额**相对 **`getReserves()`** 已有正增量。 |
| **首铸** | 增量乘积与开方后需足够大，扣掉 **`MINIMUM_LIQUIDITY`** 后仍 **`liquidity > 0`**，否则 revert。 |
| **续铸** | 两种增量与池内储备比例需匹配到你期望的程度；否则虽可能成功，但经济效果可能差。 |
| **权限** | `mint` 无白名单，任意地址可调用；谁转币谁调 `mint` 均可（常见是自己转自己调）。 |
| **Gas** | 账户有足够 **ETH**（或该链原生币）支付交易费。 |

---

## 与 Router 的关系

- **仅本仓库**：没有封装好的“一键加流动性”，需按上面步骤 **先转后 `mint`**。  
- **实际产品 / 测试网常用**：使用 **`@uniswap/v2-periphery`** 的 **`Router`**（如 `addLiquidity`），内部会处理 **最优数量、最小流动性、截止时间、ETH↔WETH** 等。  

---

## 小结

- **怎么加**：向 **Pair 地址** 转入 **token0 与 token1**，再调用 **`mint(接收LP的地址)`**。  
- **前提**：两币已入账 Pair、首铸满足 **`MINIMUM_LIQUIDITY`**、后续加池比例合理；链上 **gas** 与 **代币余额** 充足。  

---

## 脚本（本仓库）

在 **`.env`** 中配置 **`RPC_URL`**、**`PRIVATE_KEY`**、**`PAIR_ADDRESS`**，以及下面**二选一**的数量（单位均为 **wei** 整数字符串）：

| 方式 | 环境变量 |
|------|----------|
| 按 Pair 的 **token0 / token1** | **`AMOUNT0_WEI`**、**`AMOUNT1_WEI`**（可先 **`yarn check:liquidity`** 看哪个地址是 token0） |
| 按你部署时的 **TOKEN_A / TOKEN_B** | **`TOKEN_A`**、**`TOKEN_B`**、**`AMOUNT_A_WEI`**、**`AMOUNT_B_WEI`**（脚本会映射到 token0/token1） |

可选：**`LP_TO`** — 接收 LP 的地址（默认 caller）。

```bash
yarn compile
yarn add:liquidity
```

说明：脚本会发 **三笔交易**（转 token0 → 转 token1 → **`mint`**），与 Router 单笔原子 **`addLiquidity`** 不同；测试网自用足够，主网请优先使用 **`v2-periphery` Router** 并设置滑点与截止时间。
